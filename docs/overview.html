Choosing the Right AI for Our Event Chatbot

To make our event chatbot smart and helpful, we need to choose the best "brain" for it. Here's a look at different ways we can do this, with their good points and potential downsides, keeping our project goals and your needs in mind.

1. Using a "Home-Grown" AI on a Laptop

(Like our current experimental setup)

This means we run the AI software directly on a standard laptop. It's like having a small, dedicated helper working right here.

  • Your Data Stays With You: All information about events and user chats remains private on the laptop.
  • No Ongoing AI Fees: Once set up, there are no extra costs for the AI to answer questions.
  • Works Offline: The chatbot can still work even if there's no internet.
  • Great for Learning: Helps us understand how these AIs work from the inside out.
  • Limited Power: A laptop can only handle so much. This might mean slower answers or not being able to process very complex questions or lots of event details at once.
  • Can Be Slow: More advanced AI or lots of data can make it sluggish.
  • Needs Technical Setup: Requires some technical know-how to get it running and keep it updated.
  • Not for Heavy Use: Might struggle if many people try to use it at the same time.

2. Using a Paid Cloud AI Service (e.g., ChatGPT-4o)

(We still provide our scraped event data)

This option involves "renting" time on a very advanced AI, like ChatGPT, that's run by a big company. We would feed it our collected event information to answer questions.

  • Very Smart & Capable: Access to one of the best AIs available, which understands questions very well.
  • Handles Lots of Info: Can easily work with all our event details and long conversations.
  • Always On & Ready: These services are built to be reliable and handle many users.
  • No Computer Worries for Us: The AI company handles all the powerful computers needed.
  • Quick to Get Started with AI: We can focus on our chatbot's features, not the AI's plumbing.
  • Pay-As-You-Go Costs: We'd pay based on how much we use it, which could add up with many users or questions.
  • Data Goes to a Third Party: Event details and user chats are sent to the AI company's computers.
  • Relies on Another Company: We depend on their service, pricing, and rules.
  • Less "Under the Hood" Control: We can't change the core AI itself, only how we ask it questions.
  • We Still Manage Event Data: We would still need to collect and update our event information.

3. Vector Search + Embeddings with a Dedicated Database

(Our data lives in a fast search index; the LLM focuses on answers)

We turn every exhibitor page, programme item, and practical detail into numerical embeddings and store them in a vector database such as MongoDB. Each visitor question is embedded, matched to the most relevant snippets in milliseconds, and those snippets are fed to a lightweight LLM (local or cloud-hosted) to craft the reply.

  • Privacy & Control: All FFD content stays on our own server; only small context chunks reach the LLM.
  • Low Running Costs: Vector search is cheap; embeddings are computed once and reused.
  • Fast Retrieval: Sub-second answers even with tens of thousands of documents.
  • Easy to Scale: Add new editions or languages without retraining a gigantic model.
  • Model-Agnostic: We can swap the LLM (local laptop ↔ cloud API) without touching the database layer.
  • Setup Overhead: We must generate embeddings and host/maintain the vector DB.
  • DevOps Required: Backups, monitoring, and security patches for the DB.
  • Needs an LLM: Retrieval alone isn't conversational; we still run/pay a language model.
  • Content Freshness Pipeline: Website updates should trigger re-embedding.
